# `func_solve_body_monolith` playbook

The big one. **~54% of GPU time on the the production benchmark.** The single dominant kernel.

The kernel is generated by Quadrants from the `@qd.kernel` `func_solve_body_monolith` (constraint/solver.py:~3160) which inlines `func_solve_iter` (~3067) and ALL of its callees. So edits anywhere in the call graph affect this kernel.

Per `h100_vs_mi300x_kernel_comparison.txt`: 22.0 ms/call MI300X vs 6.5 ms/call H100 = 3.38× ratio. Closing that 3.38× to 1.5× saves ~10 ms/call × 511 calls = 5 seconds out of the ~17s baseline run.

## What the team already shipped

1. **AccVGPR + global_load** (reference patch): compiler-layer pass to reduce register spillage and emit `global_load` instead of `flat_load`. **9.83% E2E uplift.** Already in current Quadrants build.
2. **func_update_constraint_batch Jaref hoist** (jlohia, 0c5bd5bd): hoisted multi-read of `Jaref[i_c, i_b]` across writes to `active`. Already in current Genesis branch.
3. **monolith inner cleanup** (jlohia, bd735f8b on monolith-inner-cleanup branch): consolidated my exp 21+22 — fuse cost loop in update_constraint_batch + hoist cg_prev_Mgrad in terminate_or_update_descent_batch.

## What did NOT work (don't repeat)

| Exp | Why it failed |
|---|---|
| 17 | `block_dim=32 → 64` on the OUTER monolith loop. **+4.5% regression.** Quadrants picks smarter than naive override; AccVGPR fix tuned for 32. |
| 20 | Loop-swap to hoist `efc_force` in dense qfrc loop. **+17% regression** — turned 1 write into N_C writes; Quadrants doesn't keep arrays in registers across loops. |
| 33 | Fused dense qfrc loop with gauss/cost loop. **+1.2% regression** — fusing loops with disjoint load sets adds register pressure. |

## Tier 1: Find more "eliminate a full N_DOFS pass" wins (the exp 32 pattern)

This is the proven highest-leverage move. The Cholesky solve in `func_solve_mass_entity` was doing 3 separate passes over n_dofs (Step 1, Step 2, Step 3). exp 32 collapsed Step 2+3 into one. Look for other places where adjacent loops can fuse without increasing register pressure:

- **`func_solve_iter` qacc/Ma update + cg_prev save loops** — both iterate n_dofs, sequential. Tried in exp 24, regressed (register pressure). May be worth retrying with more careful structure.
- **`func_update_gradient_batch` grad-compute loop + Cholesky Step 1's first vec read** — cross-function fuse, eliminates an HBM round-trip on `grad`. Requires inlining `func_solve_mass_batch`'s body.
- **Cholesky Step 1 itself** — 1 outer pass with O(n²) inner work. Could the descending direction be reversed without changing math? Probably not (algorithm depends on order).

## Tier 2: More Jaref-style multi-read hoists

The team's `0c5bd5bd` Jaref hoist proved Quadrants does NOT auto-CSE across writes to other arrays. Manual hoists win here. Look for arrays read 2+ times across writes-to-other-arrays in the CG inner loop:

- `func_terminate_or_update_descent_batch`: `Mgrad[i_d, i_b]` read in fused Loop 1 AND in search update Loop 2 (post-exp28). Cross-loop multi-read. Hard to hoist without LDS.
- `func_ls_point_fn_opt`: Already heavily team-optimized (recompute quad on-the-fly).
- `func_ls_point_fn_3alphas_opt`: Same — already optimized.

## Tier 3: Reduce redundant work

Look for computations that produce values that aren't needed in all code paths:

- The CG accumulator computes `cg_beta` and `cg_pg_dot_pMg` even when `improved == False` (post-exp28 fuse). When false, this work is wasted. Could gate behind an `if improved:` check, but that defers `improved` computation. Trade-off.
- `func_solve_init`'s warmstart-pick logic (when `enable_mujoco_compatibility = True`) does duplicate Ma computation for warmstart vs current. Could reuse one if cost_ws < cost.

## Tier 4: Reduce register pressure

The team's compiler note explicitly says "Reduce Register Spillage by using AccVGPR". DSL-side, register pressure comes from:

- Many live scalars in `func_linesearch_batch` (p0_alpha, p0_cost, p0_deriv_*, p1_*, p2_*, alphas, costs, grads, hess) — 12+ live floats during Phase 3
- Hoists I add can add register pressure. Watch for regressions.

If a change reduces register pressure (e.g., reuses a variable), it could improve monolith via better occupancy.

## Tier 5 (compiler layer, OUT OF SCOPE for this loop)

- `amdgpu-waves-per-eu="1,2"` uniform constraint in Quadrants — limits monolith's occupancy
- Per-kernel per-eu attribute would let the agent pick optimal occupancy per kernel
- This is `LCOMPILER-1748`, owned by AMD compilers team

If a DSL experiment hits a register-pressure ceiling, log a `workspace/compiler_proposals.md` entry and continue with DSL.

## Anti-patterns (proven in prior sessions)

1. **Don't override `block_dim=32` on the outer monolith loop.** It's tuned. (exp 17: +4.5% regression)
2. **Don't fuse loops with disjoint load sets** — increases register pressure, hurts monolith. (exp 33: +1.2% regression)
3. **Don't loop-swap to hoist if it converts 1 write to N writes** — Quadrants doesn't keep arrays in registers across loop boundaries. (exp 20: +17% regression)
4. **Don't trust single-trial bench measurements** — ~1.5% std observed. Re-bench any apparent big win. (exp 25 false +18% turned out to be noise.)

## What "good" looks like for this campaign

monolith currently ~847 ms. Target tracker says 3520 ms total monolith time at H100, i.e., 6.9 ms/call. We're at ~22 us per i_b (847ms / ~38000 batches × 8192 ÷ 511 calls ≈ ...). The ratio is 3.38× MI300X/H100.

Closing to 2× would save ~5 seconds of e2e wall time on the production benchmark = +30% e2e improvement. That's the moonshot. Realistic this session: find 1-3 more exp-32-style wins, each shaving 0.5-1% monolith, totaling 2-3% e2e (~+1-2 pp of H100).
